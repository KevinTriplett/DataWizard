# 7.0 Safety and Recovery

Git never truly deletes anything. Every commit is a permanent snapshot. Even after a bad merge, the previous state is preserved and recoverable.

## How Git Protects You

- **Every commit is a snapshot.** You can always go back to any previous commit and see or restore every file as it was at that moment.
- **Merges are reversible.** A merge commit records both "before" states. You can undo a merge after the fact.
- **Branches are cheap bookmarks.** Creating a branch before risky work costs nothing and gives you a named save point.
- **Reflog is the last resort.** Even if you do something that seems destructive (reset, rebase, force-push), git's reflog records every state your local repo has been in for at least 30 days.

## Rolling Back a Bad Merge

Scenario: a branch was merged to main, but the changes caused problems.

### Step 1: Find the merge commit

```bash
git log --oneline --merges -5
```

You'll see something like:

```
a1b2c3d Merge branch 'restructure-2026-04-23' into main
```

### Step 2: Revert the merge

```bash
git revert -m 1 a1b2c3d
```

The `-m 1` tells git "keep main's version, undo the branch that was merged in." This creates a new commit that undoes the merge -- it doesn't erase history, it adds a correction on top.

### Step 3: Push the revert

```bash
git push origin main
```

Everyone who pulls will now get the reverted state. The merge commit and the revert commit both remain in history -- nothing is lost, and you can see exactly what happened.

## Viewing Files from Before a Merge

If you just want to see or copy files from before the merge without reverting everything:

```bash
git log --oneline -10                              # find the commit before the merge
git show <pre-merge-hash>:path/to/file.md          # view a specific file at that point
git checkout <pre-merge-hash> -- path/to/file.md   # restore a specific file
```

Or create a temporary branch to browse the entire pre-merge state:

```bash
git checkout -b pre-merge-snapshot <pre-merge-hash>
```

When done, switch back: `git checkout main`

## Restoring a Single File

```bash
git log --oneline -- "path/to/file.md"         # find the last good commit
git checkout <hash> -- "path/to/file.md"        # restore that version
```

The file is now restored in your working directory. Next auto-sync will commit the restoration.

## Undoing Your Last Commit

If you haven't pushed yet:

```bash
git reset --soft HEAD~1
```

This undoes the commit but keeps your changes as uncommitted files -- you can re-edit and recommit.

If you already pushed:

```bash
git revert HEAD
git push origin main
```

This adds a new commit that undoes the previous one. Safer than rewriting history on a shared branch.

## Rolling Back to a Safety Tag

If you created a `pre-merge-backup` tag before merging (see Section 6):

```bash
git checkout pre-merge-backup              # inspect the state
git checkout main                           # back to current main
git revert -m 1 <merge-hash>              # revert the merge (preferred)
```

Or, for a hard rollback (use with caution on shared repos):

```bash
git reset --hard pre-merge-backup
git push --force origin main
```

Force-push rewrites shared history -- only do this if all collaborators are aware and have no unpushed work.

## Nuclear Option: Full Vault Rollback

If everything is broken and you need to go back to a known good state:

```bash
git log --oneline -20                       # find target commit
git reset --hard <commit-hash>
git push --force origin main
```

This is destructive -- all changes after the target commit are permanently lost from the branch (though git reflog preserves them locally for 30 days). Only use for genuine recovery.

## Using Reflog (Last Resort)

If you've done something that seems to have lost commits:

```bash
git reflog
```

This shows every state your local repo has been in, even after resets and rebases. Find the hash of the state you want and:

```bash
git reset --hard <reflog-hash>
```

## "Oh No" Quick Reference

| Scenario | What to do |
|---|---|
| Merged a bad branch to main | `git revert -m 1 <merge-hash>` then `git push` |
| Need to see a file from before a merge | `git show <pre-merge-hash>:path/to/file.md` |
| Accidentally deleted a file and committed | `git log --oneline -- "path/to/file.md"` then `git checkout <hash> -- "path/to/file.md"` |
| Want to undo my last commit (not pushed) | `git reset --soft HEAD~1` |
| Pushed something I shouldn't have | `git revert HEAD` then `git push` |
| Everything is broken | Restore from Time Machine / File History, or `git reflog` to find a good state |
| Merge conflict I don't understand | Open the file, look for `<<<<<<<` markers, keep the version you want, delete the markers, save |
| Two people editing the same file | Communicate. If conflict happens, resolve together -- git preserves both versions |
| Created a tag before merge, now need to roll back | `git revert -m 1 <merge-hash>` (safe) or `git reset --hard pre-merge-backup` (destructive) |
