# 6.0 Branch Workflow and Pull Requests

For routine edits -- fixing a typo, adding a session log entry, updating action items -- pushing directly to main via auto-sync is fine. For larger changes, use a branch.

## When to Use a Branch

- Renaming or moving more than a handful of files
- Changing folder structure
- Bulk YAML updates across many files
- Any change you'd want someone to review before it goes live
- Experimental work you might want to discard

## Creating and Pushing a Branch

```bash
cd "/path/to/your/vault-or-project"
git checkout -b descriptive-branch-name
```

Make your changes, then:

```bash
git add -A
git commit -m "description of what changed and why"
git push origin descriptive-branch-name
```

Use descriptive branch names like `restructure-2026-05-02` or `yaml-backfill-transcripts`, not `fix` or `update`.

## Opening a Pull Request

After pushing your branch, open a Pull Request (PR) on GitHub so collaborators can review:

1. Go to your repo on GitHub
2. You'll see a banner: "descriptive-branch-name had recent pushes -- Compare & pull request"
3. Click it, add a description of what changed and why, then click "Create pull request"

If you miss the banner: go to the repo's Pull Requests tab > New Pull Request > select your branch.

## Reviewing a Pull Request

On GitHub, the PR shows every file changed as a diff -- additions in green, deletions in red.

Review at whatever depth the situation calls for:

- **Quick scan:** Skim the file list. Do the filenames make sense? Is the scope what you expected?
- **Spot check:** Click into a few changed files and read the diffs. Focus on the files that matter most.
- **Full review:** Read every diff line by line. Only needed for sensitive or structural changes.

If it looks good: click **Merge pull request** on GitHub, then **Confirm merge**.

If something needs changing: leave a comment on the PR explaining what, and the branch author can make fixes and push again.

## When You Can't Review but Need to Merge

This happens -- someone pushes a branch and the reviewer is unavailable. Options, from safest to fastest:

**Option 1: Wait.** If nothing is blocked, just wait. The branch isn't going anywhere.

**Option 2: Merge with a safety bookmark.** Before merging, create a tag at the current main so you have a named restore point:

```bash
git tag pre-merge-backup
git push origin pre-merge-backup
```

Then merge the PR on GitHub. If something breaks, you have a one-command rollback target (see Section 7).

**Option 3: Merge and trust.** If the person who made the changes is someone you trust and the scope was discussed beforehand, merging without line-by-line review is acceptable. Git's revert capability means this is never truly irreversible.

## After Merging

Once the PR is merged on GitHub, everyone's auto-sync will pull the changes within a minute or two.

If you're working in the terminal, switch back to main and pull:

```bash
git checkout main
git pull origin main
```

The branch can be deleted on GitHub (it offers this after merge) -- the changes are now part of main's history.

## Pausing the Obsidian Git Plugin During Branch Work

If you're doing branch work from the terminal but Obsidian is open, the plugin's auto-sync might try to push to the wrong branch or create auto-commits on your branch. Two options:

1. **Temporarily disable auto-sync:** Obsidian Git settings > set auto commit-and-sync interval to `0`. Re-enable when done.
2. **Close Obsidian** while doing terminal branch work, reopen when back on main.
