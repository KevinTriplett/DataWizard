# 8.0 Local Backups

Git is version control, not backup. If your hard drive fails or you accidentally delete the repo folder, git history is gone unless it's also on the remote. Keep a local backup layer running alongside git.

## macOS -- Time Machine

If Time Machine is enabled (and it should be), your vault is already backed up hourly with no action needed.

**Setup:** System Settings > General > Time Machine. Connect an external drive or use a network backup destination. Time Machine runs automatically.

**To restore a file or folder:** Open the vault folder in Finder, enter Time Machine (menu bar icon or Spotlight search for "Time Machine"), and navigate back in time to find the version you want.

**What Time Machine gives you that git doesn't:** Recovery from accidental repo deletion, recovery from a corrupted `.git` folder, and snapshots of files that git was configured to ignore (like AI plugin indexes).

## Windows -- File History

**Setup:** Settings > Update and Security > Backup > Add a drive. Select an external drive, then toggle "Automatically back up my files" to On.

For more control: click "More options" to set backup frequency (every 10 minutes to daily) and how long to keep backups.

**To restore:** Right-click the file or folder > Properties > Previous Versions tab. Select the version you want and click Restore.

**Alternative -- Windows Backup (Windows 10/11):** Settings > Accounts > Windows Backup. Backs up to OneDrive. Less granular than File History but requires no external drive.

## What About Cloud Sync (Dropbox, OneDrive, iCloud)?

Cloud sync services can conflict with git if they sync the `.git` folder. If your vault is in a cloud-synced location:

- **Best practice:** Add `.git/` to the cloud service's ignore list (Dropbox: `.dropboxignore`, OneDrive: via settings). This prevents the cloud service from syncing git internals while still syncing your actual files.
- **Alternative:** Keep the vault outside the cloud-synced folder entirely. Git + GitHub provides the offsite backup that cloud sync would otherwise give you.

Do not rely on cloud sync as your only backup -- it propagates deletions and corruptions in real time.

## The GitHub Remote as Backup

The GitHub remote itself is a backup of everything that's been pushed. If your local repo is lost entirely:

```bash
git clone https://github.com/username/repo-name.git "/path/to/vault"
```

This restores the full vault with complete history. Then reinstall Obsidian plugins and reconfigure the Git plugin (Section 3).

**Limitation:** If a bad commit was pushed to main, the remote has the bad version too. That's where git's rollback capabilities (Section 7) matter -- the bad commit is on the remote, but so is the history that lets you undo it.

## Recommended Backup Stack

| Layer | What it covers | How often |
|---|---|---|
| **Git + GitHub** | Full version history, file-level rollback, offsite copy | Every 1-5 minutes (auto-sync) |
| **Time Machine / File History** | Full system backup including non-git files, OS-level recovery | Hourly (automatic) |
| **Obsidian File Recovery** | Quick in-app single-file recovery | Every 5 minutes (snapshots) |

With all three layers, you'd have to lose your local drive, your Time Machine drive, and your GitHub account simultaneously to lose data. That's a very bad day, but not a realistic risk with basic hygiene.
