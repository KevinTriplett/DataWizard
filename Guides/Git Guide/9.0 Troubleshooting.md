# 9.0 Troubleshooting

Common issues and their fixes.

## Push Rejected: File Exceeds 100MB

GitHub rejects individual files over 100MB. This usually happens with AI plugin index files (Copilot, Smart Connections) or large PDFs.

**Step 1:** Note the filename from the error message.

**Step 2:** Add the file pattern to `.gitignore`:

```
# Example: exclude Copilot index
.obsidian/copilot-index-*.json
```

**Step 3:** The file is already in commit history, so the .gitignore alone won't help. Rewrite history to remove it:

```bash
git stash                           # save any uncommitted changes
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch "PATH/TO/LARGE/FILE"' \
  --prune-empty --tag-name-filter cat -- --all
git stash pop                       # restore uncommitted changes
git push -u origin main --force
```

If `git stash pop` fails with a "not a stash-like commit" error, the stash was rewritten along with history. This is harmless -- the stashed changes were likely minor (cursor positions, etc.). Continue without it.

## Authentication Failure on Push

GitHub no longer accepts password authentication for HTTPS.

**Option A -- Personal Access Token:**

1. GitHub > Settings > Developer settings > Personal access tokens > Tokens (classic)
2. Generate new token with `repo` scope
3. When git prompts for a password, enter the token instead

**Option B -- Switch to SSH (recommended):**

See Section 2 (SSH Keys). This is the permanent fix.

## "Git is not ready" in Obsidian Git Settings

1. Run `which git` in terminal -- note the full path
2. Obsidian Git settings > Advanced > Custom Git binary path > enter the full path
3. Click Reload
4. If it persists, fully quit and relaunch Obsidian

## Obsidian Git Not Auto-Committing

Check that:

- Auto commit-and-sync interval is set to `1` or `5` (not `0`)
- The plugin is enabled (Settings > Community Plugins)
- "Git is not ready" is NOT showing in settings
- There are actually uncommitted changes (`git status` in terminal)

## Merge Conflict on Auto-Pull

The plugin pulled changes that conflict with your local edits. The conflicted file will contain `<<<<<<<` markers.

1. Open the file in Obsidian
2. Look for the conflict markers
3. Keep the version you want, delete the markers
4. Save
5. Next auto-sync commits the resolution

If you see many conflicts at once (e.g., after being offline for a while), consider resolving them one at a time from the terminal:

```bash
git status                          # see which files are conflicted
# edit each file to resolve
git add "path/to/resolved-file.md"
git commit -m "resolve merge conflicts"
```

## zsh History Expansion Error (Mac)

If your vault path contains `!` (e.g., `_!nbox/`), zsh interprets it as a history expansion character. Wrap paths in single quotes:

```bash
cd '/Users/you/Vaults/Regen Vault/_!nbox/'
```

Or disable history expansion: `set +o histexpand`

## "fatal: not a git repository"

You're not in the right directory. Make sure you `cd` into the correct folder:

- For vault operations: `cd "/path/to/vault"`
- For Seed operations: `cd "/path/to/vault/_DataWizard/Seed"`
- For project operations: `cd "/path/to/vault/_ProjectFolder"`

Nested repos have their own `.git` -- commands must be run from inside the correct repo.

## Pull Fails: "refusing to merge unrelated histories" or Divergent Branches (Seed repo)

If this happens in `_DataWizard/Seed/`, the upstream Seed history was rewritten (this happens rarely, for security cleanups -- see the NOTICE in VERSION.md when active). Your clone and GitHub no longer share history, so pulls cannot merge.

**Do not** force the merge with `--allow-unrelated-histories` -- that reattaches the old history the rewrite was meant to remove.

Recovery:

1. Check for local-only commits: `git fetch origin && git log --oneline main --not origin/main`. If any exist, copy the affected files aside or `git stash` first.
2. Reset to the rewritten upstream: `git reset --hard origin/main`
3. `Vault Config.md` is untracked and is not affected.

If this happens in a project repo (not the Seed), stop and check with the repo owner before resetting -- project history is not expected to be rewritten.

## Push Succeeds but Collaborator Doesn't See Changes

1. Check that the collaborator's Obsidian Git plugin is running (status bar should show recent activity)
2. Have them manually pull: `Cmd+P` > `Git: Pull`
3. If still not showing: check they're on the same branch (`git branch` in terminal -- the starred branch is the current one)
4. Verify the push actually reached GitHub: check the repo's commit history on github.com

## "HEAD detached" Warning

This means you're looking at a specific commit rather than being on a branch. Common after running `git checkout <hash>` to inspect old state.

To get back to normal:

```bash
git checkout main
```

## Obsidian Shows Duplicate or Ghost Files After Git Operations

Sometimes Obsidian's file index gets out of sync with what git did (especially after branch switches or resets). Fix:

1. Close and reopen Obsidian (forces a re-index)
2. If duplicates persist, check `git status` -- the "ghost" files may be untracked leftovers. Delete them manually or `git clean -fd` (caution: this deletes all untracked files).

## Stuck Interactive Rebase

If `git status` shows "interactive rebase in progress" or you see a `rebase-merge` directory inside `.git/`, the repo is stuck mid-rebase. This can happen if a sync script or plugin used `git pull --rebase` and hit a conflict it couldn't resolve automatically. Symptoms include detached HEAD, files appearing to be missing, and commits that don't show up in the working directory.

To recover:

```bash
git rebase --abort          # abandon the rebase and return to pre-rebase state
git status                  # confirm you're back on a branch
git checkout main           # if not already on main
```

If you had uncommitted changes before the rebase started, they should be preserved. If not, check `git stash list` -- they may have been stashed automatically.

**Prevention:** Make sure your sync script uses `git pull --no-rebase` (merge strategy) rather than `git pull --rebase`. The DW sync script (`datawizard-sync.sh`) uses `--no-rebase` by default. Check your repo's git config with `git config pull.rebase` -- it should return `false` or nothing.

## .DS_Store Merge Conflicts

macOS creates hidden `.DS_Store` files in every folder you open in Finder. If these files are tracked by git, they cause recurring merge conflicts because every collaborator's macOS regenerates them with different content.

**Fix (one-time):**

1. Add `.DS_Store` to your `.gitignore` if it's not already there:

```
# macOS metadata
.DS_Store
```

2. Remove all tracked `.DS_Store` files from git (without deleting them locally):

```bash
git rm -r --cached '*.DS_Store'
git commit -m "stop tracking .DS_Store files"
git push
```

After this, macOS will continue creating `.DS_Store` files locally, but git will ignore them. Every collaborator should pull this change -- their local `.DS_Store` files will be untracked going forward.


## "Another git process seems to be running" (stale index.lock)

A commit fails with:

```
fatal: Unable to create '.../.git/index.lock': File exists.
Another git process seems to be running...
```

Git creates `.git/index.lock` while it works and removes it when done. If a git process crashes or is interrupted - or, in synced vaults, an auto-sync or a sandboxed agent leaves one behind - the lock stays and blocks every future commit.

**First, decide whether it is stale.** A lock is stale if no git process is actually running. Tell-tale signs: the lock file is old (check its timestamp with `ls -la .git/index.lock`), and commits succeeded *after* it was created (so nothing has been holding it since). If Obsidian is open, its Git plugin may be mid-sync - wait 30-60 seconds and retry before assuming it is stale.

**If it is stale, remove it and retry:**

```bash
cd "/path/to/your/repo"
rm -f .git/index.lock
git add -A && git commit -m "..." && git push
```

Removing the lock is safe *only if no git process is genuinely running.* If you are unsure, quit Obsidian first (this stops the Git plugin), then remove the lock.

## Push Rejected: "Updates were rejected" / "fetch first" (non-fast-forward)

```
! [rejected]        main -> main (fetch first)
error: failed to push some refs...
hint: Updates were rejected because the remote contains work that you do
hint: not have locally.
```

Normal and expected in a multi-machine or multi-operator setup: another machine (or another person's auto-sync) pushed to the remote after you last pulled. Your push cannot go through until you integrate their commits.

**Fix:**

```bash
git pull --no-edit && git push
```

`--no-edit` accepts the default merge-commit message so you are not dropped into a terminal text editor (vim). If the pull merges cleanly, the `&&` runs the push and you are done.

**If the pull reports a `CONFLICT`,** the same file changed on both sides. Open the conflicted file(s), resolve the marker blocks (keep the right content, delete all three marker lines), then:

```bash
git add "path/to/resolved-file.md"
git commit --no-edit
git push
```

In multi-operator projects the session log shell (`0.2`) is the most common collision point, which is why the protocol defers shell edits to session close.

## Conflict Markers That Got Committed (silent)

The dangerous case: a background auto-sync hit a merge conflict and committed the result *with the markers still in it* - so the conflict markers are now permanent content in a file, pushed to everyone.

**Why `git status` will not help:** once committed, the markers are ordinary file content, not an "unmerged" state. `git status` shows a clean tree. The only way to find them is to grep file *contents*:

```bash
grep -rl '^<<<<<<<' . --include=*.md | grep -v '/.git/'
```

**Resolve:** open each flagged file, delete the conflicting block you do not want plus all three marker lines, save, then commit the fix:

```bash
git add -A && git commit -m "resolve leftover conflict markers" && git push
```

**Watch for false positives:** files that *teach* about conflicts (this guide, onboarding docs) contain example markers on purpose. Do not "fix" those.

**Prevent it:** a `.git/hooks/pre-commit` that greps staged content for marker lines and aborts the commit stops the auto-sync from ever committing a conflicted merge. (Tracked as a feature request: [[Git Sync Failure Detector - Blue Smarties Test]].)
